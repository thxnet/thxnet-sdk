## Executive Summary

This document is the step-by-step operational runbook for validating the THXNET. blockchain upgrade from Polkadot SDK v0.9.40 to v1.12.0 across all 11 live chains (2 rootchains + 9 leafchains). The upgrade spans 30+ runtime storage migrations and 4 critical upstream backports cherry-picked after the `polkadot-v1.12.0` tag was cut. The strongest motivation for this plan is the **2026-04-11 incident**: a leafchain became stuck at a session boundary (block #7182000) after an improperly sequenced upgrade, with the collator panicking in `ParachainSystem::on_finalize` because the `set_validation_data` inherent was rejected by `CheckWeight`. The P0 fix (`6d83f4bb06`) is now included, but this plan exists to prove the fix works on live state before touching mainnet. Success criteria: all 11 chains survive the upgrade with continuous block production, finalization, and correct session boundary crossings, validated on testnet before mainnet.

## Section 1: Testnet 4-Step Upgrade Dry Run

**Priority: MUST DO -- blocks mainnet upgrade**

This section replicates the exact upgrade sequence that will be applied to mainnet, but on testnet first. The upgrade order is: rootchain binary -> leafchain binary -> rootchain WASM -> leafchain WASM. This is the correct order because binary upgrades are backward-compatible with old runtimes, but new runtimes may require new host functions only available in new binaries.

> **Lesson from the 2026-04-11 incident:** The incident used the sequence rootchain binary -> leafchain binary -> rootchain WASM -> leafchain WASM and the leafchain got stuck 18 blocks after the WASM upgrade at a session boundary. The root cause was a missing upstream backport (P0: check-weight mandatory PoV exemption). This plan validates that the fix works.

### Phase 3: Rootchain WASM Upgrade (MUST DO)

**Objective:** Submit the new rootchain testnet runtime (`thxnet_testnet_runtime.compact.compressed.wasm`) via `system.setCode`. This triggers all rootchain runtime storage migrations (30+ migrations spanning v0.9.40 -> v1.12.0).

**Estimated time:** 15 minutes (submission + migration execution) + 30 minutes (observation through at least 1 session boundary)

**Risk level:** HIGH -- runtime migrations are irreversible. Once `system.setCode` is enacted, the storage is migrated forward. Downgrading the runtime typically requires a matching reverse migration, which does not exist.

```bash
    | jq -r '.result.isSyncing // "unknown"')

  echo "[$(date '+%H:%M:%S')] Block: #${BEST}, Peers: ${PEERS}, Syncing: ${SYNCING}"

  # Detect stuck sync (no progress in 10 minutes)
  if [ "${BEST}" = "${PREV_BEST}" ]; then
    STUCK_COUNT=$((STUCK_COUNT + 1))
    if [ ${STUCK_COUNT} -ge 10 ]; then
      echo "FAIL: Sync stuck at #${BEST} for 10+ minutes"
      break
    fi
  else
    STUCK_COUNT=0
  fi
  PREV_BEST="${BEST}"

  # Detect completion
  if [ "${SYNCING}" = "false" ] && [ ${BEST} -gt 0 ]; then
    echo "Sync complete at #${BEST}"
    break
  fi

  sleep 60
done
```

**Priority: FUTURE**

This section outlines the phased plan for automating the validation steps above into CI, so future upgrades do not require manual execution.

### Phase 0: Fix try-runtime create-snapshot call site bug

**Estimated effort:** 1-2 hours

**What it catches:** Enables idempotency testing, which is currently silently skipped.

**The bug:** The `try-runtime-test.sh` script uses `--create-snapshot` as a flag on the `on-runtime-upgrade live` subcommand. However, in try-runtime v0.10.1, `create-snapshot` is a **top-level subcommand** (not a flag). The script's `snapshot_supported()` function checks for `--create-snapshot` in the help text of `on-runtime-upgrade live`, does not find it, and silently skips all idempotency tests.

**Fix location:** `scripts/try-runtime-test.sh`

```bash
# Current (BROKEN):

#   "${TRY_RUNTIME_BIN}" --runtime "${wasm_path}" \

# Fixed: Use create-snapshot as a top-level subcommand
#   "${TRY_RUNTIME_BIN}" create-snapshot \
#     --uri "${uri}" \
#     --snapshot-path "${snapshot_file}"
```

The fix requires:
1. Change `snapshot_supported()` to check for `create-snapshot` as a subcommand
2. Change all `create_snapshot()` invocations to use the subcommand form

### Phase 3 (Deferred): Zombienet state-restore binary swap

**Why it is hard:** Zombienet currently does not support restoring from a live chain state snapshot. It always starts from genesis. To test binary swap upgrades, we would need to:
1. Export state from a live node
2. Modify the chain spec to include the exported state as genesis
3. Start Zombienet with the modified spec
4. Swap binaries and observe behavior

This is fragile and time-consuming. Zombienet is better suited for genesis-based testing (e.g., "does the new binary start and produce blocks from genesis?") than for upgrade-path testing.

**Timeline:** Defer until Zombienet supports state import natively, or until we build custom tooling.

### Runtime (WASM) Downgrade Limitations

**Runtime downgrades are generally NOT possible** without a reverse storage migration.

When `system.setCode` is enacted:
1. The new runtime executes `on_runtime_upgrade()`, which runs all pending migrations
2. Storage is modified (keys renamed, formats changed, new keys added)
3. StorageVersion values are bumped
4. The old runtime expects the old storage format

If you submit the old WASM via `system.setCode`:
- The old runtime will attempt to read storage in the old format
- Storage keys that were renamed or reformatted will appear missing
- The chain will likely panic or produce incorrect state

**Emergency options if runtime upgrade fails:**
1. **Hotfix WASM:** Build a new WASM that fixes the issue and submit via `system.setCode`
2. **State revert (relay chain only):** If >2/3 of validators agree, revert to a pre-upgrade state backup. This is a governance decision.
3. **Fork and restart (parachain):** Coordinate with relay chain governance to deregister and re-register the parachain with corrected state

### Emergency Contact Protocol

If the upgrade fails and manual intervention is needed:
1. Immediately capture logs from all affected nodes
2. Record the exact block height where failure occurred
3. Check if the failure matches known patterns (Appendix D)
4. If the chain is stuck, do NOT restart nodes -- the state may be diagnostic

## Appendix D: Session Boundary Monitoring

### What happened in the 2026-04-11 incident

1. The leafchain upgraded to spec version 17 from spec version 3
2. ~18 blocks were produced successfully after the WASM upgrade
3. At session 1996 boundary (block #7182000), block production stopped
4. The collator panicked with:

```
thread 'main' panicked at '`set_validation_data` inherent needs to be present
in every block!', cumulus/pallets/parachain-system/src/lib.rs
```

**Root cause:** Without the P0 fix (`d3bf1a82ac`), `CheckWeight::pre_dispatch` enforced the total PoV (Proof of Validity) size limit on ALL extrinsics, including mandatory inherents like `set_validation_data`. At session boundaries, the PoV is larger because it includes new session keys and authority changes. When the PoV exceeded the limit, `CheckWeight` rejected the `set_validation_data` inherent, causing it to be excluded from the block. `ParachainSystem::on_finalize` then panicked because it expected `set_validation_data` to always be present.

The P0 fix (`6d83f4bb06`) exempts mandatory extrinsics from the PoV size check, so `set_validation_data` is never rejected regardless of PoV size.

### How to detect the incident pattern early

Monitor collator logs for these patterns:

```bash
# Pattern 1: The exact panic
grep -iE 'set_validation_data inherent needs to be present' <LOG_FILE>

# Pattern 2: PoV size exhaustion (precursor)
grep -iE 'ExhaustsResources|exhausts_resources' <LOG_FILE>

# Pattern 3: Generic validation data issues
grep -iE 'ValidationData' <LOG_FILE>

# Pattern 4: Generic panics
grep -iE 'thread.*panicked at' <LOG_FILE>

# Pattern 5: Block production stall (check relay chain)
grep -iE 'candidate_backing|parachain.*stall|no.*candidate' <RELAY_LOG_FILE>
```

### How to know when a session boundary is approaching

Relay chain session boundaries are deterministic:

```bash
# Get current block number
CURRENT=$(curl -s -H "Content-Type: application/json" \
  -d '{"id":1,"jsonrpc":"2.0","method":"chain_getHeader","params":[]}' \
  https://node.testnet.thxnet.org/archive-001/ws \
  | jq -r '.result.number' | xargs printf "%d")

# Get epoch duration (session length) via runtime constant
# Typically 600 blocks (1 hour at 6-second blocks) for testnet
# Check via: Polkadot.js Apps -> Developer -> Chain State -> babe -> epochDuration
SESSION_LENGTH=600  # Adjust based on actual configuration

BLOCKS_IN_SESSION=$((CURRENT % SESSION_LENGTH))
BLOCKS_TO_BOUNDARY=$((SESSION_LENGTH - BLOCKS_IN_SESSION))
SECONDS_TO_BOUNDARY=$((BLOCKS_TO_BOUNDARY * 6))
MINUTES_TO_BOUNDARY=$((SECONDS_TO_BOUNDARY / 60))

echo "Current block: #${CURRENT}"
echo "Blocks into current session: ${BLOCKS_IN_SESSION}/${SESSION_LENGTH}"
echo "Blocks to next session boundary: ${BLOCKS_TO_BOUNDARY}"
echo "Estimated time to boundary: ~${MINUTES_TO_BOUNDARY} minutes"
```
